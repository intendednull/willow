# Willow-Channel Removal — Implementation Plan

**Date**: 2026-04-12
**Spec**: `docs/specs/2026-04-12-willow-channel-removal.md`
**Depends on**: `docs/plans/2026-04-12-state-authority-and-mutations.md`
(permission pre-check must be in place before mutations are rewritten)

## Overview

Remove `willow-channel` and make `willow-state::ServerState` the
client's sole source of truth. Work is ordered to keep compilation
passing after each step.

## Step 1: Add `ChannelKind` to willow-state

**File**: `crates/state/src/types.rs`

Add the enum:

```rust
#[derive(Clone, Debug, PartialEq, Eq, Serialize, Deserialize)]
pub enum ChannelKind {
    Text,
    Voice,
}
```

Update `Channel` to use it:

```rust
pub struct Channel {
    pub id: String,
    pub name: String,
    pub pinned_messages: BTreeSet<EventHash>,
    pub kind: ChannelKind,  // was String
}
```

**File**: `crates/state/src/event.rs`

Update `EventKind::CreateChannel` to use the enum:

```rust
CreateChannel {
    name: String,
    channel_id: String,
    #[serde(default)]
    kind: ChannelKind,  // was String
},
```

Add a `Default` impl for `ChannelKind` returning `Text` (for serde
backward compat, replaces `default_create_channel_kind()`).

**File**: `crates/state/src/materialize.rs`

Update `apply_mutation` for `CreateChannel` to use `kind` directly
(no longer a String).

**File**: `crates/state/src/lib.rs`

Export `ChannelKind` from the crate root.

**Tests**: Existing state tests should still pass. Update any test
that constructs `EventKind::CreateChannel` with `kind: "text".into()`
to use `ChannelKind::Text`.

Run `cargo test -p willow-state`.

## Step 2: Move `Invite` types to willow-client

**Files**:
- `crates/client/src/invite.rs` — add `InviteId(Uuid)` and `Invite`
  struct here (moved from `crates/channel/src/lib.rs`).
- Remove the `willow_channel::Invite` and `InviteId` imports.

Update `generate_invite()` signature — instead of taking
`&willow_channel::Server`, take the data it needs directly:

```rust
pub fn generate_invite(
    server_name: &str,
    server_id: &str,
    channels: &[(String, String)],  // (topic, channel_name)
    keys: &HashMap<String, ChannelKey>,
    recipient_ed25519_public: &[u8; 32],

```

Update the 4 invite tests to construct state via `ManagedDag` instead
of `willow_channel::Server::new()`.

Run `cargo test -p willow-client`.

## Step 3: Rewrite client mutations to use event pipeline

**Prerequisite**: The permission pre-check from the authority plan must
be in place, so `create_and_insert` rejects unauthorized events before
signing.

### 3a: `mutations.rs` — `create_channel`

Replace `entry.server.create_channel(name, ChannelKind::Text)` with
`build_event(EventKind::CreateChannel { ... })`. The channel ID is a
new UUID string generated by the caller.

### 3b: `actions.rs` — `create_voice_channel`

Same pattern: replace `entry.server.create_channel(name, Voice)` with
`build_event(EventKind::CreateChannel { kind: ChannelKind::Voice, ... })`.

Remove the `willow_channel::ChannelKind` imports, use
`willow_state::ChannelKind`.

### 3c: `actions.rs` — role operations

Replace `willow_channel::RoleId(Uuid::parse_str(...))` with plain
`String` role IDs, since state uses strings.

### 3d: `mutations.rs` — role creation

Replace `willow_channel::RoleId::new()` and
`willow_channel::Role::with_id()` with
`build_event(EventKind::CreateRole { ... })`.

Run `cargo test -p willow-client`.

## Step 4: Remove `Server` from `ServerEntry`

**Files**: `crates/client/src/state_actors.rs`, `crates/client/src/state.rs`

Remove the `server: willow_channel::Server` field from `ServerEntry`.

Remove `topic_map: HashMap<String, (String, willow_channel::ChannelId)>`.
Derive topic-to-channel mapping from `ServerState.channels` instead
(a helper function or computed on access).

### 4a: `util.rs`

Change `make_topic()` from:
```rust
pub fn make_topic(server: &willow_channel::Server, channel_name: &str) -> String
```
to:
```rust
pub fn make_topic(server_id: &str, channel_name: &str) -> String
```

Update all call sites.

### 4b: `views.rs`

Replace `willow_channel::ChannelKind::Text` / `::Voice` pattern
matches with `willow_state::ChannelKind::Text` / `::Voice`.

Read channel list from `ServerState.channels` instead of
`entry.server.channels()`.

### 4c: `persistence_actor.rs`

Change `PersistServerConfig` and `PersistServerById` to hold
server data from `ServerState` fields, not `willow_channel::Server`.

### 4d: `storage.rs`

Update serialize/deserialize to work with `ServerState` instead of
`willow_channel::Server`.

### 4e: `lib.rs`

Delete `reconcile_topic_map()`. Delete `parse_permission()` (use
`willow_state::Permission` directly).

Run `cargo test -p willow-client`.

## Step 5: Rewrite invite join path

**File**: `crates/client/src/joining.rs`

The join path currently constructs a `willow_channel::Server` from
invite data:

```rust
let mut server = willow_channel::Server::with_id(
    willow_channel::ServerId(parsed_server_uuid),
    &accepted.server_name,
    accepted.genesis_author,
);
```

Replace with seeding `ServerState` via genesis event and channel
creation events through `ManagedDag`, or by directly constructing a
minimal `ServerState` with the invite's channel list and waiting for
full state sync from peers.

Run `cargo test -p willow-client`.

## Step 6: Rewrite tests

**File**: `crates/client/src/lib.rs`

The `test_client()` helper constructs a `willow_channel::Server`. It
must be rewritten to seed state through `ManagedDag`:

```rust
fn test_client() -> (ClientHandle<MemNetwork>, ...) {
    // ... create identity, ManagedDag ...

    // CreateChannel event creates "general"
    // State is now populated via the event pipeline
}
```

Update tests that reference `willow_channel` types:
- `send_message_and_read_back`
- `create_channel_shows_in_list`
- Invite-related tests

**File**: `crates/client/src/invite.rs`

Rewrite the 4 test functions to construct servers via `ManagedDag`
instead of `Server::new()`.

Run `cargo test -p willow-client`.
